全流程：

在 Java 中，编译器（javac）只负责把 .java 翻译成半成品 .class（字节码），然后就丢给 JVM 去解释执行了。

预处理（Preprocessing）、编译（Compilation）、汇编（Assembly） 和 链接（Linking）。

1. 预处理 (Preprocessing)

  • 输入：.cpp 源文件 和 .h 头文件
  • 核心任务：处理所有以 # 开头的指令，纯做文本的增删替换。
    • 复制粘贴：将 #include 的头文件内容全部原封不动地拷贝进来。
    • 宏替换：将 #define 定义的宏替换为实际的数值或代码。
    • 去注释：剥离代码中的所有注释内容。
  • 输出：.i 文件（体积庞大的纯 C++ 源码文件）

2. 编译 (Compilation)

  • 输入：.i 文件
  • 核心任务：将高级语言降维成底层硬件语言。
    • 词法/语法分析：严格检查 C++ 语法是否合法（绝大多数的语法报错在此阶段产生）。
    • 代码优化：剔除永远不会执行的死代码，优化计算逻辑（如 constexpr 的预计算就是在这里完成）。
    • 翻译：将 C++ 翻译成特定 CPU 架构（如 x86、ARM）能懂的汇编代码。
  • 输出：.s 文件（汇编语言源文件）

3. 汇编 (Assembly)

  • 输入：.s 文件
  • 核心任务：将汇编语言翻译成纯粹的机器语言。
    • 机械翻译：把人类还能勉强看懂的汇编指令（如 MOV, ADD），逐条翻译成 CPU 唯一能执行的 0 和 1 二进制机器码。
  • 输出：.obj (Windows) 或 .o (Linux) 文件（包含了机器指令的目标文件碎片）

4. 链接 (Linking)

  • 输入：所有单独编译出来的 .obj 碎片 + 系统库或第三方库文件
  • 核心任务：地址寻址与物理缝合。
    • 符号决议：解决“未解析的外部符号”。根据头文件提供的声明，去其他 .obj 碎片里寻找函数的真正实现，填补内存地址。
    • 合并封装：将所有零散的二进制碎片和调用的外部库，像乐高一样严丝合缝地拼装在一起。
  • 输出：.exe (Windows) 或 .elf (Linux) 文件（最终可双击运行的程序）

空间分配

1. 代码区

  • 具体工作： 存放 CPU 正在执行或准备执行的机器指令（也就是你写的 C++ 代码编译后的二进制码）。
  • 特性： 它是只读的（防止程序意外篡改自己的指令），并且是共享的（如果你同时开两个相同的仿真程序，它们在内存里只共用这一份代码指令，节省空间）。

2. 常量区

  • 具体工作： 专门存放程序中无法修改的常量数据，最典型的就是字符串字面量和某些被 const 修饰的全局变量。
  • 特性： 同样是绝对只读。如果试图通过强转指针去修改这里的值，程序会直接崩溃（段错误 Segmentation Fault）。

3. 全局/静态区

  • 具体工作： 存放所有的全局变量和被 static 修饰的变量（无论是全局静态还是局部静态）。
  • 特性： 这个区域的生命周期与整个程序“同生共死”。程序启动时它就在，程序结束时它才销毁。在 Java 中，静态变量存在方法区（Method Area / Metaspace），概念非常类似。
    • 细分说明：在底层，它通常又被细分为已初始化数据段（.data）和未初始化数据段（.bss，操作系统会自动将其清零）。

4. 堆

具体工作： 提供动态内存分配。当你觉得栈空间不够大，或者需要一个对象的生命周期跨越多个函数调用时，就在这里申请空间。

  • 特性： 空间极大（通常受限于物理内存大小），由程序员通过 new/malloc 手动申请，必须通过 delete/free 手动释放。如果你忘记释放，就会造成内存泄漏。堆的分配地址通常是向上增长的（向高地址方向）。

5. 栈

  • 具体工作： 维持函数调用的上下文。专门存放函数内部的局部变量、函数的形参、以及函数执行完的返回地址。
  • 特性： 极速但容量极小（在 Windows 系统下通常默认只有 1MB 左右，Linux 下默认 8MB）。由编译器全自动管理，函数执行完立刻自动释放，绝不发生内存泄漏。栈的分配地址通常是向下增长的（向低地址方向）。

二.

命名冲突与名称简介

命名空间是一个区域，在其中声明名称以消除歧义。命名空间为其内部声明的名称提供了一个作用域区域（称为命名空间作用域）——意味着在命名空间内声明的任何名称都不会与其他作用域中的相同名称混淆。

在同一命名空间中，所有名称必须唯一，否则将导致命名冲突。

std命名空间

std::cout实际上就是cout，而std是标识符cout所属的命名空间名称。因为cout是在std命名空间中定义的，所以名称cout不会与在全局命名空间中创建的名为cout的对象或函数冲突。

类似地，当访问在命名空间中定义的标识符（例如std::cout）时，需要告诉编译器我们要找的是在命名空间（std）内定义的标识符。

头文件是怎么将两个.cpp文件结合到一起的？

一开始只是加了引用，在链接阶段才找到函数的主体.cpp文件

独立编译（cpp到compile 两条线）：

  • 左边：add.cpp 拿着自己的代码，可能也引用了 add.h 核对了一下接口，然后独自被编译器（Compile）变成了机器码碎片 add.o（obj）。
  • 右边：main.cpp 里面调用了 add.cpp 写的函数。但是编译器每次只能看一个文件，它根本不知道 add.cpp 的存在。此时，#include "add.h" 发挥作用了。它把函数的“声明（长什么样）”拷贝到了 main.cpp 里。编译器一看：“哦，虽然我不知道具体怎么算，但格式对得上，放行！”于是 main.cpp 也独自被编译成了 main.o。

物理缝合（图中的 Link 节点）：

  • 真正把两个文件联系在一起的，是底部的 链接器（Linker）。
  • 它就像一个拼图大师，把 add.o（真实的函数逻辑）、main.o（调用者的指令）以及标准库（如 iostream 底层的 Standard Runtime Library）硬生生地拼接在一起，最终输出了完整的 main.exe。

为避免函数不匹配，包含add方法的add.cpp文件都要添加上#include"add.h"

#pragma once

避免头文件中被其他头文件包含后，头文件中的内容被重复编译，导致多个相同函数，添加#pragma once使得函数不会在其他头文件中在此编译。（该语法来自现代编译器）

五

常量const

const 只是编译器给你加的一把“权限锁”，它决定了你能不能修改这块内存

全局常变量 / 静态常变量 —— 全局生命周期

如果你在所有函数之外定义了一个常变量，或者在函数内部用 static 修饰了它，它的生命周期就是整个程序的运行期。

  • 存储位置： 常量区（.rodata）或全局/静态区。
  • 生命周期： 在 main() 函数开始执行前就已经分配好内存并初始化，直到程序完全关闭时才被系统销毁。

2. 局部常变量 —— 栈的生命周期

这是最容易被误解的一种。如果你在一个普通函数内部定义了一个普通的 const 变量，它根本不会进入常量区，而是直接被压入【栈区】。

  • 存储位置： 栈区（Stack）。
  • 生命周期： 当代码执行到这一行时诞生，当执行流程离开它所在的大括号 {} 时，它会随着栈帧的弹出而瞬间灰飞烟灭。

constexpr 关键字

constexpr（是“constant expression”的缩写）

当使用 const 关键字声明 const 变量时，编译器会隐式地跟踪它是运行时常量还是编译时常量。在大多数情况下，这个区别除了影响优化之外并不重要，但在某些场景下 C++ 需要常量表达式，而常量表达式中只能使用编译时常量。

由于编译时常量还允许更好的优化（且几乎没有缺点），因此我们通常希望尽可能多地使用编译时常量,尽早在编译阶段就发现问题。

使用 const 时，变量到底是编译时常量还是运行时常量，取决于其初始值是否为编译时常量表达式。

constexpr使得变量只能是编译时常量

任何在初始化之后不会修改、并且初始值在编译时就能确定的变量，都应声明为 constexpr。

任何在初始化之后不会修改、但初始值在编译时无法确定的变量，则应声明为 const。

与 constexpr 不兼容的类型（包括 std::string、std::vector 等使用动态内存分配的类型）。对于这些类型的常量对象，请改用 const。

std::string和std::string_view

相当于java的string，但是需要引入string库，进行标准化输出std::string

当你需要一个只读字符串时，尤其是作为函数参数时，优先使用 std::string_view 而非 std::string。

十一

模板

在C++中，模板系统旨在简化创建能够处理不同数据类型的函数（或类）的过程。

template <typename T> T get_max(...) 本身并不是一个函数。它只是一张图纸，是不占用代码区（Code Segment）内存的。

当你真正在代码里调用 get_max(10, 20) 时，它悄悄地帮你真正写出了一个新的函数，并把它编译进了最终的机器码里。这个过程叫做模板实例化。
